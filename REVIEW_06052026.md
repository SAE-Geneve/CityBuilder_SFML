# CityBuilder_SFML — Code Review (2026-05-06)

> *"Don't tell me your code works. Tell me how it can break."* — in the spirit of Jason Turner

This pass focuses strictly on **undefined behavior, lifetime bugs, indexing
mistakes, silent overallocation, and pessimizations the optimizer cannot
rescue you from**. Style nits, naming, and architectural taste are left
out — the project already has `GUIDELINES.md` and `FORBIDDEN_FEATURES.md`
for those, and several findings below cite them because the tree is in
violation of its own rules.

Severity tags:
- **[UB]** — undefined behavior or guaranteed wrong result
- **[LEAK]** — owned memory escapes the program
- **[ALLOC]** — avoidable allocation, often per-frame or per-NPC
- **[PERF]** — pessimization the optimizer will not undo
- **[BUG]** — wrong but well-defined behavior
- **[RULES]** — violates the repo's own `GUIDELINES.md` / `FORBIDDEN_FEATURES.md`

## 1. `core/include/ai/a_star.h` — non-injective index mapping **[UB-adjacent / BUG]**

```cpp
const int width  = static_cast<int>(tilemap.extent(0));
const int height = static_cast<int>(tilemap.extent(1));
auto to_index = [width](Vec2i p) { return p.x * width + p.y; };
...
trace = {idx / width, idx % width};
```

`p.x` ranges over `[0, width)`, `p.y` ranges over `[0, height)`. The
index formula `p.x * width + p.y` is only injective when `p.y < width`.
The moment `height > width`, two distinct cells produce the same index,
`came_from`/`g_cost`/`visited` are aliased, and the algorithm returns
silently incorrect paths (or none). The recovery `{idx/width, idx%width}`
amplifies this — `idx % width` can never produce a `y` in `[width, height)`.

Either index by `p.y * width + p.x` (the conventional row-major form,
which *is* injective because `x < width`), or compute against
`max(width, height)` consistently. The existing tests use a 3×3 map and
will never catch this.

While here: `to_index` is captured by `[width]`, but the recovery line
hard-codes the same `width` again. If you ever flip to `height`, you
will only fix one of the two. Capture both, or compute one from the
other.

## 2. `api/src/motion/a_star.cc` — raw `new`, leaks, dangling chains **[LEAK][RULES]**

```cpp
//FIXME when is this "new aStarNode(currentNode)" getting deleted?
aStarNode newNode =
    aStarNode(newPosition, currentNode.g + 1,
              heuristic(newPosition, end), new aStarNode(currentNode));
```

The `FIXME` answers itself: it never gets deleted. Every neighbor
expansion of every node leaks one `aStarNode`. For a 25×20 map with
moderate fan-out you can leak thousands of nodes per `GetPath` call.
This also directly violates `FORBIDDEN_FEATURES.md` ("no raw
`new`/`delete`").

The "right" fix is not `unique_ptr` — pointer-chasing a heap chain to
reconstruct a path is already wrong. Store nodes in a single `std::vector<aStarNode>`
keyed by index (the `core/include/ai/a_star.h` template *almost*
does this — it stores `came_from[]` indices), and reconstruct via
indices. No allocations per neighbor, no leaks, better cache behavior,
and `previous_node` becomes an `int` you can `static_assert` is
trivially copyable.

While here:

- `Path GetPath(... std::vector<sf::Vector2f> walkableTiles)` takes the
  walkables **by value**. Every call site (`NpcBehaviourTree::SetDestination`)
  hands it `tilemap_->GetWalkables()`, which copies the entire walkable
  set per pathfind. Take `std::span<const sf::Vector2f>`.
- `std::ranges::find(walkableTiles, newPosition)` is `O(N)` and is run
  once per neighbor of every popped node. With hundreds of walkables and
  hundreds of nodes, you have an accidental `O(N²)`. A `std::unordered_set`
  keyed by quantized position, or — simpler — query the tilemap directly
  by `(x/step, y/step)`, fixes this.
- `closedList` is `std::vector<aStarNode>` searched with `find_if`. Same
  issue, same fix: a `std::vector<bool>`-equivalent (use
  `std::vector<std::uint8_t>`, **never** `std::vector<bool>` per the rules)
  indexed by tile.
- `neighbours()` returns a fresh `std::array` on every call. Make it
  `static constexpr` at namespace scope; it's pure data.
- The open list never updates a node when a cheaper `g` is found —
  duplicates accumulate. With the heuristic mismatched to step cost
  (`g + 1` vs Euclidean `h`), this is currently producing a non-admissible
  search.
- Float equality comparisons on `sf::Vector2f` (`currentNode.position == end`,
  `newPosition == n.position`) only happen to work because everything is
  generated from integer multiples of `gridStep`. Quantize to `Vec2i`
  internally and stop relying on it.
- `#include <filesystem>` in `a_star.h` — completely unused. Drop it; it
  is one of the heaviest standard headers.

## 3. `api/src/motion/path.cc` — `IsDone()` on empty path **[BUG]**

```cpp
bool Path::IsDone() const {
  return idxPoint_ == path_points_.size() - 1;
}
```

When `path_points_` is empty, `size() - 1` is `static_cast<size_t>(-1)`
(well-defined unsigned wraparound, but **not** what you want). The
comparison promotes `idxPoint_` (int) to `size_t`; on an empty path,
`IsDone()` returns `false` forever. Today every caller pre-checks
`IsValid()`, so this is latent — but it is a hand grenade with the pin
out. Two-line fix:

```cpp
return !path_points_.empty() && idxPoint_ + 1 == std::ssize(path_points_);
```

`std::ssize` is already endorsed by `GUIDELINES.md`.

While here: `Points()` returns `std::vector<sf::Vector2f>` **by value**
and is called from `NpcBehaviourTree::SetDestination`:

```cpp
this->path_->Fill(path.Points());
```

That is one full copy out of `Astar::ReconstitutePath`, one `Path::Points()`
copy, then one more copy inside `Path::Fill`. Three copies of the same
path. Make `Points()` return `const std::vector<sf::Vector2f>&` (and
mark it `const`), or have `Fill` take the path-producing object directly.

## 4. `api/src/graphics/tilemap.cc` — three indexing mistakes **[BUG][BUG][BUG]**

```cpp
void TileMap::SetTile(int idx, Tile tile) {
  if (idx > 0 && idx < tiles_.size()) { ... }
}
```

`idx > 0` excludes tile 0. Should be `idx >= 0`. As a bonus, the
comparison `idx < tiles_.size()` is signed-vs-unsigned; warnings-as-errors
should already be biting here unless they were silenced. `std::cmp_less`
or `std::ssize` resolves both.

```cpp
int TileMap::Index(const sf::Vector2f screenPosition) {
  return static_cast<int>(ceil(screenPosition.y / kPixelStep * kWidth)) +
         static_cast<int>(ceil(screenPosition.x / kPixelStep));
}
```

`kWidth` is in **pixels** (1600). The row stride is `kWidth / kPixelStep`
(25). For any `y > 0`, this returns an index well off the end of `tiles_`.
The function is currently unused publicly (it is `static` and only
referenced inside the file), which is the only reason this hasn't caused
a crash.

```cpp
sf::Vector2f TileMap::TilePos(sf::Vector2i pos) {
  return {static_cast<float>(round(pos.x / kPixelStep) * kPixelStep),
          static_cast<float>(round(pos.y / kPixelStep) * kPixelStep)};
}
```

`pos.x / kPixelStep` is **integer division**. `round()` of an integer is
a no-op promoted through `double` and back. The function is intended to
snap to the grid centroid; instead it floors. Either cast the numerator
to `float` first, or — simpler — `(pos.x / kPixelStep) * kPixelStep`
without the `round` and call it what it is.

```cpp
sf::Vector2f TileMap::ScreenPosition(const int index) {
  float x = ceil((index % (kWidth / kPixelStep)) * kPixelStep);
  float y = ceil((index / (kWidth / kPixelStep)) * kPixelStep);
  return {x, y};
}
```

Same problem inverted: integer math fed to `ceil`, which then does
nothing. Drop `ceil`. Mark the function `constexpr`. It is pure data
shuffling.

## 5. `api/src/graphics/tilemap.cc` — Draw allocates and rebinds 500 sprites/frame **[ALLOC][PERF]**

```cpp
//FIXME use sf::VertexArray instead of sf::Sprite
sf::Sprite sprite(textures_.Get(Tile::kEmpty));
sprite.setTextureRect(sf::IntRect({0, 0}, {kPixelStep, kPixelStep}));
for (auto tile : tiles_) {
  sprite.setTexture(textures_.Get(tile));
  sprite.setPosition(ScreenPosition(tileIndex));
  window.draw(sprite);
  tileIndex++;
}
```

The `FIXME` is right. 500 draw calls per frame against per-tile texture
binds defeats GPU batching. A single `sf::VertexArray` (or an atlas plus
one draw) is what every SFML tilemap tutorial since 2012 has shown. This
is the single biggest CPU/GPU win available to this project.

Also: the loop variable `tileIndex` is `int`; `tiles_.size()` is `size_t`.
Use a structured binding over `std::views::enumerate`, or just
`std::ssize(tiles_)` and compare apples to apples.

## 6. `api/include/resources/resource_manager.h` — every NPC gets its own copy of the world **[ALLOC][BUG]**

```cpp
inline std::vector<Resource> ResourceManager::GetResources(
    const Resource::Type type) const {
  std::vector<Resource> resources_of_type = {};
  for (const auto& resource : resources_) {
    if (resource.GetType() == type) {
      resources_of_type.emplace_back(resource);
    }
  }
  return resources_of_type;
}
```

This is called from `npc_factory.h::CreateNpc`:

```cpp
npcs.back().Setup(type, "...", tilemap, start_position,
                  std::move(resource_manager.GetResources(Resource::Type::kWood)));
```

Two problems, one cosmetic, one fundamental:

1. **[PERF]** `std::move` on the result of a function that already returns
   a prvalue blocks RVO and accomplishes nothing. The C++23 rules give
   you the move for free; the explicit `std::move` here is a Jason Turner
   slide title.
2. **[BUG]** Each NPC stores its own *copy* of every wood/stone/food
   `Resource`. When NPC A chops a tree, only A's copy decrements;
   NPCs B, C, D still see the original quantity. The `ChopEvent` callback
   does set the tile to background, but the `Resource::quantity_` state
   is per-NPC. So 12 NPCs chopping the same forest will collectively
   harvest 12× the wood that exists. This is invisible until you spawn
   enough NPCs (which the new `--init-spawn` flag now does for you).

The fix is not "share a `vector<Resource>` by reference" — the lifetime
is fine, but the *identity* model is wrong. A `Resource` should live in
exactly one place (the `ResourceManager`), and NPCs should reference it
by index/handle. Then `Exploit` mutates the canonical copy and everyone
sees it.

While at it: `inline` definitions of non-trivial functions in the
header violate `GUIDELINES.md` ("no in-header function bodies except
trivial/template").

## 7. `api/include/ai/bt_action.h`, `clickable.h`, etc. — `std::function` everywhere **[RULES][ALLOC]**

`FORBIDDEN_FEATURES.md` lists `std::function` explicitly. The project
already noticed this in `resources/resource.h` (there's a comment
swapping out `std::function` for a function pointer). The same surgery
is owed in:

- `core/include/ai/bt_action.h` — `std::function<Status()> action_;` —
  one heap allocation per behaviour-tree leaf, per NPC, today. The
  `TODO` admits it. Replace with either CRTP, a function pointer +
  context, or a small templated `Action<F>` so the lambda is inlined.
- `api/include/ui/clickable.h` — `OnReleasedLeft`, `OnReleasedRight`,
  `OnPressedLeft`, `OnPressedRight`, `OnHoverEnter`, `OnHoverExit` are
  six `std::function<void()>` per `Clickable`. Most are unset most of
  the time, so the small-buffer optimization mostly hides the cost —
  but the moment a captured lambda exceeds the SBO threshold (e.g. the
  `[this]` capture in `Button`'s ctor compiles to one pointer, fine; a
  larger capture would silently allocate). Either define a virtual
  interface, or template the type.

## 8. `api/include/ai/npc.h` — every NPC owns its own behaviour tree **[ALLOC]**

```cpp
/* FIXME each npc has its own bt_tree that allocates which is the same...
   What about 4 bt_trees (one for each role) shared between all NPC */
std::unique_ptr<NpcBehaviourTree> bt_tree_ =
    std::make_unique<NpcBehaviourTree>();
```

Spawn 100 NPCs and you build 100 identical trees. Each tree is, in turn,
a tree of `unique_ptr<Node>` with `std::function`-based `Action` leaves.
Allocations cascade.

The `NpcBehaviourTree` mixes two concerns that should be separated:

- **State** — `hunger_`, `current_ressource_`, `path_` pointer, `tick_dt`.
  Keep this per NPC.
- **Topology** — the `Selector { Sequence{...}, Sequence{...}, Idle }`.
  This is identical across NPCs of a given role. Build it once.

Once split, the topology can be `static constexpr` (or at least built
once at startup) and the per-tick `Tick()` takes a reference to the
state. Memory allocations drop to one per role rather than one per NPC.

While here: also a default member initializer for `texture_`:

```cpp
std::unique_ptr<sf::Texture> texture_ = std::make_unique<sf::Texture>();
```

Every freshly default-constructed NPC heap-allocates an empty `sf::Texture`,
even though `Setup` is going to overwrite it via `loadFromFile`. Remove
the default init and have `Setup` do the construction. Or hold the
texture by value.

## 9. `core/include/maths/vec2.h` — divide-by-zero in `Normalize` **[UB-adjacent]**

```cpp
Vec2& Normalize() {
  const auto length = Length();
  x /= length;
  y /= length;
  return *this;
}
```

Dividing a `float` by `0.f` is not technically UB in IEEE-754 — it
yields ±Inf or NaN — but it propagates through every downstream
calculation. The function is currently unused, but it's a footgun
sitting on the table. Either return `std::optional<Vec2>` (the rules
prefer `std::expected` for error paths and `std::optional` only for
optional values; "the unit vector of zero" is genuinely optional →
optional is fine), or assert.

Also: `Length()` calls `std::sqrt` and is not `noexcept`. `SqrLength()`
is the right primitive for hot paths; document that.

## 10. `api/include/motion/motor.h` — `normalized()` of a zero vector **[UB-adjacent]**

```cpp
inline void Motor::Update(const float dt) {
  const sf::Vector2f distance = destination_ - position_;
  remainingDistance_ = distance.length();
  if (remainingDistance_ < speed_ * dt) {
    position_ = destination_;
    return;
  }
  position_ += distance.normalized() * speed_ * dt;
}
```

SFML 3 documents `Vector2::normalized()` as having a precondition that
the vector is non-zero (it asserts in debug, returns `{nan, nan}` in
release). The early-return guards against `remainingDistance_ < speed_*dt`
— but if `speed_` is zero (initial state until `SetSpeed` is called) or
`dt` is zero (paused frame), the guard collapses to `remainingDistance_
< 0`, which is false; `normalized()` runs anyway, on a possibly-zero
vector. Also relevant: `clock.restart()` in `Loop` is **not** called
before `Setup()`, so the first frame's `dt` will reflect setup time —
typically positive and large, not zero, but the contract should hold
regardless. Guard explicitly:

```cpp
if (remainingDistance_ <= std::numeric_limits<float>::epsilon()) return;
```

## 11. `api/src/ui/clickable.cc` — unconditional dereference of `std::optional` **[UB]**

```cpp
void Clickable::HandleEvent(std::optional<sf::Event> evt, bool &wasClicked) {
  const auto released = evt->getIf<sf::Event::MouseButtonReleased>();
```

`evt->...` on a disengaged optional is UB. Today, every caller in
`game.cc` is inside `while (const std::optional event = window_.pollEvent())`,
so `evt` is engaged. But the function is `public` and the parameter is
literally `std::optional<sf::Event>` — the API is *advertising* that
empty is acceptable. Either:

- Take `const sf::Event&` and put the engagement check at the call site
  (preferred — the optionality lives in `pollEvent`'s return, not in
  `HandleEvent`'s contract), or
- Add `if (!evt) return;` at the top.

The current shape lets a future caller hand in `std::nullopt` and crash
with optimizations on. While here: the optional is taken **by value** —
`sf::Event` is a `std::variant` of all SFML event types and is not free
to copy. Pass by const reference.

Cosmetic-but-jarring: `wasClicked |= true;` in four places. That is
`wasClicked = true;` written by someone who wanted to feel clever. Pick
one.

## 12. `game/src/game.cc` — namespace-scope statics with side effects **[FRAGILE]**

```cpp
auto tilemap_ptr_ = std::make_unique<TileMap>();
api::ai::NpcManager npc_manager_;
api::ui::ButtonFactory btn_factory;
ResourceManager resource_manager;
```

`btn_factory`'s constructor calls `font.openFromFile("_assets/fonts/ANTQUAB.TTF")`
during static initialization. That runs before `main`, before any
`chdir`, before SFML has been initialized. It currently works because
nothing else touches the filesystem during static init *and* the working
directory happens to be the build dir. Move all of this into `Setup()`,
where the order is explicit and you can return errors. This also fixes
the `std::cout << "Failed to load font" << std::endl;` (use
`core::LogError`) inside the factory's ctor — there is nothing useful a
caller can do with that message during static init.

While here:

- `btnExit = btn_factory.CreateButton(sf::Vector2f(window_.getSize().y - 30.f, 30.f), "Exit");`
  uses `getSize().y` for the **X** coordinate. The button currently lands
  at `(1080-30, 30)` instead of `(1280-30, 30)`. With a square-ish window
  this looks "fine" until someone resizes.
- `clock` is constructed at namespace scope and never `restart()`'d
  before the game loop. The first `dt` will include all of `Setup()` —
  whatever NPCs you've spawned will teleport on frame 1.
- `Loop` returns no error code; `main` returns 0 unconditionally. If
  `window_.create` fails, you'll just spin a closed window in the loop
  guard.

## 13. `api/include/ai/npc_factory.h` — emplace + `back()` is fine, but the redundancy isn't **[STYLE/BUG]**

```cpp
switch (type) {
  case NpcType::kBlueWood: npcs.emplace_back(); npcs.back().Setup(...); break;
  case NpcType::kRedRock:  npcs.emplace_back(); npcs.back().Setup(...); break;
  case NpcType::kGreenFood:npcs.emplace_back(); npcs.back().Setup(...); break;
  default: core::LogError("Error: Npc type not found"); break;
}
if (type == NpcType::kBlueWood || type == NpcType::kRedRock || type == NpcType::kGreenFood) {
  npcs.back().SetPosition(start_position);
}
```

The trailing `if` re-checks the same three enumerators that the switch
already handled. Move `SetPosition(start_position)` into each `case` and
delete the `if`. (Note: `Npc::Setup` already takes `cantina_position`
but **not** the NPC's spawn position — that's why `SetPosition` is
needed externally. Consider making `Setup` take both.)

`#include <iostream>` at the top of this header is unused (the only
streaming line is commented out) and pulls in massive translation-unit
weight wherever this header is included.

## 14. `api/include/ai/npc_behaviour_tree.h` — `tick_dt` member **[SMELL]**

```cpp
float tick_dt = 0;
```

Set by `Update(dt)`, read by `Eat()`, `GetRessource()`, `Idle()`. This
is "passing an argument via a member variable" — once you fix the
shared-tree issue (item 8), `Tick()` should take a context with `dt`
and any other per-frame inputs. Today, if any leaf calls back into the
tree (it doesn't, but nothing prevents it), `tick_dt` is a hidden
side-channel.

While here: `kHungerRate = 2.f` is the rate at which hunger **goes
down** in `Eat()` and **goes up** in `GetRessource()` and `Idle()`. They
share the same constant, which is almost certainly not what was
intended — eating and starving at the same rate makes the simulation
neutral.

## 15. `core/include/ai/bt_node.h` — move ctor swaps a single int **[STYLE]**

```cpp
Node(Node&& node) noexcept { std::swap(status_, node.status_); }
```

`status_` is an enum, trivially copyable. `swap` of two ints is the same
as a copy + assignment — no advantage. The whole rule-of-five block can
be `= default` for move and `= delete` for copy, which is what the
comment promises ("movable-only").

## 16. `core/include/assets/asset_manager.h` — out-of-range read before bounds check **[UB]**

```cpp
void Load(std::span<const std::string_view> assetsPaths) {
  for (size_t i = 0; i < assets_.size(); ++i) {
    assets_[i] = TAsset(std::format("{}/{}", folderPath.str, assetsPaths[i]));
  }
}
```

The loop iterates up to `assets_.size()` (which is
`static_cast<size_t>(TAssetType::kLength)`), but indexes `assetsPaths[i]`
without checking that the span is at least that long. The current call
sites (`tilemap.cc`, `button_factory.h`) happen to pass arrays sized to
match `kLength`, but the function signature accepts a `span` — by
construction a runtime-sized object — and silently reads off the end if
it's shorter. Either `assert(assetsPaths.size() >= assets_.size())` at
the top, or take `std::span<const std::string_view, N>` with a
non-dynamic extent and let the type system enforce it.

`Get()` does the right thing (throws on out-of-range), so the policy is
inconsistent.

## What I am **not** flagging (but you might think I should)

- The `Composite::children_` `std::vector<std::unique_ptr<Node>>` is
  fine. Each tree is built once; the indirection is paid back by virtual
  dispatch needing a stable address.
- `tiles_` as `std::array` is correct and cache-friendly.
- The behaviour-tree state machine has bugs (see #14) but the *shape*
  (Selector → Sequence → Action) is sound for this scale.
- `Vec2<T>::operator==` does float-equality for floating-point `T`. Not
  great in general, but the current code never compares vectors that
  came from independent computations, so it's defensible. Document the
  contract.
- `Resource::on_chop_resource_` as a raw function pointer is correct and
  preferable to `std::function`. Keep it.

## Suggested order of attack

1. **Fix the leaks and the index bugs first** (items 2, 3, 4) — those
   produce wrong answers, not just slow ones.
2. **Fix the per-NPC resource copy** (item 6) — this is a gameplay bug
   the new `--init-spawn` flag will start surfacing immediately.
3. **`std::function` → templates / function pointers** (items 7, 8) —
   removes the worst per-NPC allocation pattern and unblocks the
   "shared behaviour tree" refactor.
4. **`sf::Sprite` per-tile → `sf::VertexArray`** (item 5) — biggest
   single-frame win.
5. Everything else is polish.
